00

What is OOP?

Before learning the syntax, understand the mindset. OOP is not a feature of C++ — it is a completely different way of thinking about programs.

The Big Idea

Programs as collections of objects

In traditional programming, a program is a list of instructions — do this, then do that, call this function, loop over this array. Everything is driven by the sequence of steps.

OOP changes that idea. Instead of thinking in steps, you think in things. Every real-world entity that your program cares about becomes an object. A Student is an object. A Car is an object. A Bank Account is an object. Each object knows its own data and knows what it can do.

Your program then becomes the story of how these objects interact with each other — a Student enrolls in a Course, a Customer purchases a Product, a Car accelerates. The logic lives inside the objects themselves, not spread across many separate functions.

💡
OOP is a programming paradigm — a style or methodology. It is not unique to C++. Java, Python, and many other languages support it. C++ supports both procedural and object-oriented styles in the same program.
OOP vs Procedural

How the two styles compare

Procedural thinking OOP thinking
What steps do I follow? → What objects are involved?
Functions operate on data passed in → Objects carry their own data and functions
Data is shared, anyone can change it → Data is protected inside each object
Code becomes harder to manage as the program grows → Objects are reusable and easier to organise
What Every Object Has

Data + Operations — always both

Every object in OOP is made of two things. Take a real-world Student:

Data — what it knows
name, studentId, GPA, studyLevel
These describe the object's current state
Operations — what it can do
enroll(), updateGPA(), getInfo()
These define the object's behaviour

In code, data becomes data members and operations become member functions. Together they form a class — the blueprint for that type of object.

The Four Pillars

What OOP is built on

OOP has four core principles. You will study all of them — the first two are covered in this course:

🔒
Encapsulation
Bundling data with the functions that use it, and hiding internal details
🧬
Inheritance
A class can inherit properties and behaviour from another class
🎭
Polymorphism
The same function name can behave differently depending on the object
📦
Abstraction
Exposing only what is necessary, hiding the complex implementation inside
📍
This guide covers Encapsulation in depth. The others build on top of it and come in later courses.
?
Think About It
Why does it matter that OOP models programs as objects rather than steps?
Because real-world problems are naturally described in terms of things and their relationships — not sequences of instructions. A hospital system has Patients, Doctors, Appointments. Writing them as objects makes the code closer to real life, which makes it easier to understand and update when something changes.
01

Structures

Before classes existed, structs were how C++ grouped data. Understanding them makes classes much easier to grasp.

The Core Idea

What is a struct?

A struct lets you bundle multiple related variables into one named unit. Instead of having ten separate variables for one employee, you create one Employee type that holds them all.

  • Members can be different types — int, string, float, even arrays
  • Once defined, you use it like any built-in type (int, float…)
  • Each variable you create from a struct gets its own independent copy of all members
Defining & using a struct
struct Employee {
    int    id;
    string name;
    float  salary;
    float  bonus;
};

int main() {
    Employee emp1;          // one Employee variable
    emp1.id     = 101;
    emp1.name   = "Sara";
    emp1.salary = 8000;
    emp1.bonus  = 1500;

    cout << emp1.name << " earns: " << emp1.salary + emp1.bonus;
    // Output: Sara earns: 9500
}
💡
The dot operator (.) is how you reach inside a struct variable to get or set a member. Think of it as saying: "go inside emp1, then find salary." It is used the same way for both structs and class objects.
?
Think About It
What if you didn't have structs — how would you store data for 50 employees?
You'd need 50 separate variables for each piece of data — id1, id2… name1, name2… salary1, salary2… That's hundreds of unconnected variables with no clear relationship. A struct packages all of one employee's data together, making your code clean and easy to manage.
Going Further

Arrays & Nesting

Structs support two useful features: storing an array as a member, and putting one struct inside another.

Array inside struct
Store multiple values of the same kind inside one member
e.g. float marks[3] — three subject marks per student
Struct inside struct
Use another struct as a member type
e.g. a Student has an Address which is its own struct
Both patterns together
struct Date { int day, month, year; };

struct Student {
    string name;
    float  marks[3];   // array member
    Date   birthDate;  // nested struct
};

Student s;
s.name       = "Ali";
s.marks[0]   = 85;
s.marks[1]   = 90;
s.birthDate.year = 2003;  // two dots for nested access
02

Struct Operations

Two things you can do with struct variables — and one thing you absolutely cannot.

Assignment

Copying one struct into another

You can assign one struct variable to another of the same type using =. This copies every member at once — no need to copy field by field.

Assignment — works fine
Employee emp1, emp2;
emp1.id = 5;  emp1.name = "Mona";  emp1.salary = 7000;

emp2 = emp1;  // copies ALL fields of emp1 into emp2

cout << emp2.name;   // → Mona  (emp2 is now an independent copy)
Comparison — Read Carefully

You cannot use == on structs

C++ does not support comparing two struct variables directly with ==. You must compare each member individually.

Comparison — right vs wrong
// ✘ THIS DOES NOT WORK — will cause a compile error
if (emp1 == emp2) { ... }

// ✔ THIS IS CORRECT — compare member by member
if (emp1.id   == emp2.id   &&
    emp1.name  == emp2.name &&
    emp1.salary == emp2.salary) {
    cout << "Same employee";
}
⚠️
Common exam trap. You can assign structs with = but you cannot compare them with ==. Always compare member by member.
?
Think About It
Why do you think C++ allows = on structs but not ==?
Assignment (=) is simple — just copy all values from one variable to the other. But equality (==) is more complex: should two structs be "equal" if only some members match? What if a float member has a small difference? C++ avoids deciding that for you. You decide what "equal" means by writing the comparison yourself.
Functions

Passing structs to functions

A struct variable can be passed by value (a copy is made) or by reference (the original is used). Functions can also return a struct.

Function that takes and returns a struct
struct Point { float x, y; };

Point midpoint(Point a, Point b) {
    Point mid;
    mid.x = (a.x + b.x) / 2;
    mid.y = (a.y + b.y) / 2;
    return mid;
}

Point p1 = {0, 0};
Point p2 = {4, 6};
Point m  = midpoint(p1, p2);   // m.x = 2, m.y = 3
03

Classes & Objects

A struct groups data. A class groups data and the functions that work on it — and controls who can touch what.

?
Before You Read
If a struct can already group data, why do we need classes at all?
A struct just holds data — anyone can read or change it freely. A class adds control: it can hide data, enforce rules (like "salary can't be negative"), and bundle the operations that belong to that data alongside it. Classes also naturally scale to larger programs where multiple developers need a safe contract for how data is used.
The Blueprint

What is a Class?

A class is a blueprint. It defines what data every object of that type will have, and what operations it can perform. The class itself is not an object — it's just a description.

A simple class
class Car {
  private:
    string brand;
    int    speed;

  public:
    void setBrand(string b) { brand = b; }
    void accelerate()        { speed += 10; }
    void printInfo()         { cout << brand << " @ " << speed << "km/h"; }
};
💡
Data members are the variables (brand, speed). Member functions are the actions (setBrand, accelerate). Together they define what a Car is and what it can do.
The Instance

What is an Object?

An object is a real, living instance created from a class. Every object gets its own independent copy of the data members. You can have as many objects as you need from the same class — each one is separate.

Creating and using objects
Car car1;              // object 1 — has its own brand & speed
Car car2;              // object 2 — completely independent

car1.setBrand("Toyota");
car1.accelerate();    // car1.speed is now 10

car2.setBrand("Honda");
// car2.speed is still 0 — separate from car1

car1.printInfo();     // → Toyota @ 10km/h
car2.printInfo();     // → Honda @ 0km/h
Class
The blueprint — defined once
Like a house plan — it's not a real house
Object
A real instance in memory
Like a house built from that plan — real and usable
04

Encapsulation

One of the four pillars of OOP. The idea is simple: hide the data, control how it is accessed.

?
Why Does This Exist?
What if anyone in the program could directly change an object's data? What could go wrong?
Imagine a BankAccount object where anyone could write account.balance = -9999. There's no rule stopping it. With encapsulation, the balance is private — no one outside the class can touch it directly. The only way in is through a controlled method (like deposit()) which can validate the amount first.
The Concept

Wrapping data + protection together

Encapsulation has two parts working together:

  • Bundling — data and the functions that operate on it live in the same class
  • Hiding — the internal data is declared private, invisible to outside code

The result is that the only way to interact with an object's data is through its public functions — a safe, controlled interface.

🔒
Data Hiding is the direct outcome of encapsulation. Think of it like a vending machine — you press a button (public method), and something happens inside (private data changes). You never reach inside the machine directly.
05

Access Specifiers

Three keywords that control exactly who can see and touch each member of a class.

The Three Levels

public, private, protected

SpecifierInside the classDerived classAnywhere outside
public✔ Yes✔ Yes✔ Yes
protected✔ Yes✔ Yes✘ No
private✔ Yes✘ No✘ No

The most used pattern: data members → private, functions → public.

Typical class structure
class BankAccount {
  private:
    double balance;      // hidden — only the class can touch this
    string ownerName;

  public:
    void   deposit(double amount);  // everyone can call this
    double getBalance();
};
?
Notice This
What's the difference between struct and class when it comes to access?
Only one difference: in a struct, members are public by default. In a class, they are private by default. That's literally the only built-in distinction in C++. Everything else — constructors, functions, inheritance — works the same in both.
🗂️
UML diagrams (the box diagrams in your lectures) use – to mean private and + to mean public. e.g. – balance : double and + getBalance() : double.
06

Getters & Setters

The standard way to give controlled access to private data — without exposing it directly.

?
The Key Question
If data is private, how does anything outside the class ever use it?
Through public functions that act as a safe doorway. A getter lets you read the value. A setter lets you write it — but only after the setter validates the input. The data never leaves the protection of the class.
Getter + Setter

Controlled read and write

Setter
Sets a private member's value
Can validate the input before accepting it
Getter
Returns a private member's value
Read-only — the original data stays safe
Setter with validation — the real benefit
class Product {
  private:
    string name;
    float  price;

  public:
    // Setter — rejects negative prices
    void setPrice(float p) {
        if (p >= 0)
            price = p;
        else
            cout << "Price cannot be negative!";
    }

    // Getter — safe read access
    float  getPrice()           { return price; }
    string getName()            { return name; }
    void   setName(string n)    { name = n; }
};
✅
If price were public, any code could write product.price = -500 and there's nothing you can do. The setter catches that attempt and rejects it.
07

Function Overloading

Same name, different job. C++ picks the right version automatically based on what you pass in.

The Idea

One name, multiple versions

You can define several functions with the same name as long as their parameter lists differ — either in the number of parameters, or their types. The compiler decides which version to call.

Overloaded functions
int   add(int a, int b)              { return a + b; }
float add(float a, float b)          { return a + b; }
int   add(int a, int b, int c)       { return a + b + c; }

add(2, 3);           // → uses int version       → 5
add(1.5f, 2.5f);     // → uses float version     → 4.0
add(1, 2, 3);        // → uses 3-parameter version → 6
⚠️
Changing only the return type does NOT count as overloading. The parameters must be different. The compiler uses the parameters to decide which version to call — not the return type.
?
Why Bother?
Why not just give each function a unique name — addInts, addFloats, addThree?
You could — but the caller now has to remember which name to use depending on their data type. With overloading, you always just write add() and C++ chooses the right version. It makes the code simpler and easier to read.
08

Constructors

A special function that runs automatically the moment an object is created — before you even touch it.

?
Why Do We Need This?
What happens if you create an object and forget to initialise its data before using it?
The data members would contain random leftover values from memory — not zero, not empty, just whatever was stored there before. Imagine creating a BankAccount and the balance starts as some random number like -837261. A constructor guarantees that every object starts with known, valid values — automatically, without you having to remember to set anything.
Rules of Constructors

What makes a constructor special

1
Same name as the class
If the class is called Car, the constructor is also called Car
2
No return type — not even void
This is how the compiler recognises it as a constructor
3
Called automatically
Runs the instant you write Car myCar; — you never call it manually
4
Usually public
So that code outside the class can create objects
Three Types

Default, Parameterised, Copy

All three constructors in one class
class Rectangle {
  private:
    float width, height;

  public:

    // 1. DEFAULT — no arguments, sets safe starting values
    Rectangle() {
        width = 1;  height = 1;
    }

    // 2. PARAMETERISED — caller provides the values
    Rectangle(float w, float h) {
        width = w;  height = h;
    }

    // 3. COPY — creates a new object from an existing one
    Rectangle(const Rectangle &src) {
        width  = src.width;
        height = src.height;
    }

    float area() { return width * height; }
};

Rectangle r1;            // default → 1×1
Rectangle r2(5.0, 3.0); // parameterised → 5×3
Rectangle r3 = r2;      // copy → 5×3 (independent copy)
Default Constructor
No arguments — safe fallback values
If you write no constructor at all, C++ provides this but it does nothing useful
Copy Constructor
Initialises from another object
C++ provides one automatically. The copy is independent — changing one won't affect the other
💡
Constructors support an initialisation list shorthand — Rectangle(float w, float h) : width(w), height(h) {} — which does exactly the same thing but more efficiently.
09

Destructor

The mirror of the constructor — runs automatically when an object's life ends.

The Concept

Cleaning up when an object is done

When a variable goes out of scope — like when a function ends — any objects created inside it are destroyed. The destructor runs at that moment, giving you a chance to release resources.

  • Name: same as class, but with a ~ prefix — e.g. ~Rectangle()
  • No parameters, no return type
  • Only one per class — it cannot be overloaded
  • You rarely write one unless your class allocates memory manually
Seeing the object lifecycle
class Session {
  public:
    Session()  { cout << "Session started\n"; }
   ~Session() { cout << "Session ended\n";   }
};

int main() {
    cout << "Before block\n";
    {
        Session s;            // → "Session started"
    }                          // s goes out of scope → "Session ended"
    cout << "After block\n";
}

// Output:
// Before block
// Session started
// Session ended
// After block
?
Real-World Use
When would you actually need to write a destructor?
Mainly when your constructor uses new to allocate heap memory. Without a destructor to call delete, that memory stays occupied even after the object dies — a memory leak. For simple classes with only basic data types, C++ cleans up automatically and you don't need to write one.

Static Members

Most data belongs to objects. Static data belongs to the class itself — one shared copy, no matter how many objects exist.

?
Picture This
How would you count how many Student objects have been created throughout a program?
You can't use a regular data member — each object has its own copy, so they can't share a count. You need a static member — one variable that all objects read from and write to the same location. Every time a constructor runs, it increments the shared counter.
Static Data + Static Functions

Shared across all objects

  • Declared with static — only one copy exists regardless of how many objects are created
  • Must be defined outside the class body: int Student::count = 0;
  • Accessed via the class name: Student::count (no object needed)
  • Static functions can only access other static members — they have no this pointer
Counting created objects
class Student {
  private:
    string name;
    static int count;   // one copy, shared by all

  public:
    Student(string n) : name(n) { count++; }
   ~Student()                   { count--; }

    static int getCount() { return count; }
};

int Student::count = 0;   // define outside the class

Student s1("Ali"), s2("Mona"), s3("Layla");
cout << Student::getCount();   // → 3
💡
Notice Student::getCount() uses the class name, not an object. This is how you call a static function — no object is needed.
11

Splitting a Class into Files

Real projects separate the what from the how. The header declares; the .cpp defines.

?
Why Split?
Why not just put everything in one file?
You can — for small programs. But as projects grow, splitting files keeps things more organised. It also allows you to share just the header file (which shows what the class can do) without sharing the actual code inside the functions.
Three Files

The standard project structure

📄
Rectangle.h — the declaration
Contains only the class structure: member names, types, and function prototypes. No function bodies.
⚙️
Rectangle.cpp — the implementation
Contains the actual code for each function. Uses ClassName:: to link back to the class.
🚀
main.cpp — your program
Just #include "Rectangle.h" and use the class freely.
Rectangle.h
#pragma once
class Rectangle {
  private:
    float width, height;
  public:
    Rectangle();
    Rectangle(float w, float h);
    float area();
};
Rectangle.cpp
#include "Rectangle.h"

Rectangle::Rectangle()
    : width(1), height(1) {}

Rectangle::Rectangle(float w, float h)
    : width(w), height(h) {}

float Rectangle::area() {
    return width * height;
}
📌
The :: scope resolution operator is what links the function definition back to its class. Without it, C++ would treat it as a regular function that has nothing to do with the class.
